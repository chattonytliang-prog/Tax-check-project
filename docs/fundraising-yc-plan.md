# 融资与 YC 取向改进计划

更新时间：2026-06-23

本文档面向项目从“税务风险报告 MVP”走向可融资产品的下一阶段规划。目标不是继续堆功能，而是把产品改造成更符合 YC 和早期资本偏好的高频、刚需、可规模化、可形成数据和工作流护城河的业务。

## 一、融资判断

当前项目已经具备演示价值：能录入企业财税画像、命中风险规则、生成税务风险体检报告，并初步接入账号、D1、规则库和 AI 报告能力。

但从融资视角看，当前仍更像“AI 税务风险报告生成器”。这个定位容易被投资人理解为咨询辅助工具、单次报告工具或小型 SaaS，融资吸引力有限。

更适合的融资叙事是：

```text
面向财税服务机构和多实体企业的 AI 税务合规操作系统，
自动读取发票、流水、申报表、工资社保和平台收入数据，
持续发现涉税异常，生成证据链、整改任务和审计底稿。
```

核心变化：

- 从“生成报告”变成“持续税务风控工作流”。
- 从“手工录入”变成“多源数据自动导入和交叉验证”。
- 从“AI 润色报告”变成“AI 税务分析助理 / agent”。
- 从“规则演示”变成“可版本化、可审计、可复核的专业规则资产”。

## 二、目标客户聚焦

不建议第一阶段同时面向所有企业、集团、税务顾问和小微企业。

建议优先聚焦：

```text
财税服务机构 / 代理记账公司 / 税务顾问团队
```

理由：

- 单个机构可以覆盖几十到几百家企业，获客效率高于逐个卖给小微企业。
- 税务风险初筛和报告生成是重复性工作，具备高频使用场景。
- 客户有明确付费能力：节省人力、降低复核成本、提升交付物质量。
- 可以沉淀跨行业、跨客户的规则反馈和风险样本。

第一阶段建议再选一个更窄行业作为 wedge，例如：

- 直播电商 / MCN
- 商贸流通
- 制造企业
- 跨境电商
- 医美 / 教培等高合规压力行业

## 三、产品改进方向

### 1. 数据导入优先于继续扩表单

当前手工录入适合演示，不适合真实规模化使用。下一阶段优先支持：

- Excel / CSV 导入
- 发票明细导入
- 银行流水导入
- 纳税申报表导入
- 工资社保表导入
- 电商 / 直播平台收入导入
- 后续扩展 ERP、账套、RPA 或第三方财税系统连接

投资人更关心系统是否能自动处理真实数据，而不是表单字段是否完整。

### 2. AI 从报告润色升级为税务分析 agent

当前 AI 能力主要用于报告优化和数据复核。融资版本应升级为：

- 自动识别资料缺失
- 自动抽取关键税务字段
- 自动匹配风险规则和政策依据
- 自动解释异常原因
- 自动生成整改任务
- 自动追问缺失材料
- 自动形成审计底稿
- 人工复核后反哺规则和模型

目标是替代一部分初级税务分析师的重复工作，而不是只把报告写得更好看。

### 3. 建立可信规则库

当前规则库可用于 MVP 验证，但正式融资需要把规则做成专业资产。

每条规则建议补齐：

- 政策依据
- 依据链接或出处
- 适用地区
- 适用行业
- 适用纳税人类型
- 生效时间
- 版本号
- 专业复核人
- 触发样例
- 误报 / 漏报反馈
- 变更记录

长期护城河应来自：

```text
规则体系 + 真实数据样本 + 专家反馈 + 工作流沉淀
```

### 4. 工作流化而不是报告化

融资版本应让用户围绕“风险整改”持续使用，而不是只生成一次报告。

建议补齐：

- 风险事项列表
- 整改任务
- 责任人
- 截止日期
- 资料补充状态
- 复核结论
- 风险关闭记录
- 月度 / 季度复查
- 企业级风险看板

这样产品才能形成留存和复购。

## 四、商业模式建议

不建议把主模型设计为“单份报告收费”，这容易变成项目制或服务制。

建议优先验证：

### 财税机构版本

- 按客户企业数收费
- 例如：每企业每月固定费用
- AI 分析和报告作为用量包
- 高级行业规则包单独收费

### 企业版本

- 按主体数和模块收费
- 适合多实体集团、连锁企业、平台型企业
- 年费制更适合融资叙事

### 专业规则包

- 行业专项包
- 地区政策包
- 税种专项包
- 稽查风险专项包

目标是形成 ARR，而不是一次性报告收入。

## 五、验证指标

融资前 6-8 周建议拿到这些证据：

- 访谈 20 家以上财税机构或税务顾问团队。
- 拿到 3-5 家付费试点。
- 每家试点机构导入 20-100 家企业资料。
- 每周有重复使用。
- 至少 1 家客户愿意预付。
- 每份初筛从数小时降到 10-20 分钟。
- 形成 3 个以上真实风险发现案例。
- 明确客户愿意为企业数、报告数或 AI 分析量付费。

融资材料里，真实牵引力比功能列表更重要。

## 六、90 天冲刺路线

### 第 1-2 周：客户发现

- 明确目标 ICP。
- 访谈 20 家财税机构 / 税务顾问。
- 找出最高频、最痛、最愿意付费的场景。
- 确定一个行业 wedge。

### 第 3-4 周：真实数据导入

- 支持 Excel / CSV 导入。
- 支持至少一种发票或流水格式。
- 能对真实资料自动生成初步风险结果。
- 输出资料缺失清单。

### 第 5-6 周：工作流闭环

- 增加风险事项、证据链、整改任务和复核状态。
- 支持人工确认、误报标记和规则反馈。
- 生成更正式的客户交付报告。

### 第 7-8 周：付费试点

- 拿到 3-5 家付费试点。
- 记录使用频次、节省时间、风险发现数量和客户反馈。
- 固化试点报价。

### 第 9-12 周：融资材料

- 准备 1 分钟 demo。
- 准备 YC 申请回答。
- 准备 traction 数据。
- 准备客户案例。
- 准备产品路线图和市场切入策略。

## 七、YC 申请叙事

建议一句话定位：

```text
AI-native tax risk analyst for accounting firms,
turning messy financial documents into audit-ready risk findings and remediation workflows.
```

中文版本：

```text
面向财税服务机构的 AI 税务风险分析师，
把发票、流水、申报表和工资社保等杂乱资料，
自动转化为可复核的风险事项、证据链和整改工作流。
```

建议强调的非显而易见洞察：

```text
税务监管正在从“以票管税”走向“以数治税”。
企业和财税机构手上有大量发票、流水、申报、工资、平台收入数据，
但缺少像税局一样进行交叉验证的能力。
本产品把这种交叉验证能力产品化，先服务财税机构，再沉淀行业风险模型。
```

## 八、近期优先级

1. 锁定 ICP，不再泛化面向所有企业。
2. 做数据导入，不再主要依赖手工录入。
3. 做风险整改工作流，不只生成报告。
4. 把规则库做成专业资产。
5. 拿真实付费试点。
6. 用使用数据和客户案例准备融资，而不是只展示 demo。

## 九、成功标准

进入融资状态前，至少应达到：

- 有明确目标客户和高频使用场景。
- 有真实数据导入能力。
- 有 3-5 个付费试点。
- 有可量化效率提升。
- 有客户愿意持续使用。
- 有规则库和数据反馈形成护城河的路径。
- 有清晰 ARR 模型。

达到以上状态后，项目才更像 YC 和早期资本会认真看的 AI-native B2B 软件公司。
