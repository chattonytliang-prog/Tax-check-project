# 上线待办清单

更新时间：2026-06-23

本文档整理当前项目从 MVP / 内部演示走向正式上线前需要补齐的事项。优先级按上线风险排序，建议先完成 P0 后再开放真实用户试用。

## 当前状态

- 前端生产构建可以通过。
- 当前架构为 Vite React 前端、Cloudflare Pages Functions API、Cloudflare D1 数据库。
- 已有基础登录、用户数据隔离、管理员后台、规则库维护和 DeepSeek 服务端调用。
- 当前仍更适合内部演示或小范围试点，不建议直接公开商用。

## P0：正式上线阻塞项

- [ ] 关闭或限制公开注册。
  - 现状：登录页直接提供注册入口，`/api/auth/register` 可被任意访问。
  - 建议：改为管理员邀请、白名单注册或审批后开通。

- [ ] 给登录、注册、AI 报告接口增加限流。
  - 现状：关键接口缺少 rate limit，存在撞库、刷账号和刷 AI 成本风险。
  - 建议：使用 Cloudflare Turnstile、WAF 规则或 Workers KV / D1 / R2 之外的专门限流方案。

- [ ] 生产环境隐藏服务端错误详情。
  - 现状：`serverError` 会把 `detail` 返回给前端，AI 请求失败也会返回部分上游响应。
  - 建议：前端只返回通用错误码和 request id，详细错误写入服务端日志。

- [ ] 修复普通用户只检测部分规则的风险。
  - 现状：非管理员从 `/api/rules` 只能拿到前 5 条规则；前端拿到 managed rules 后会只用这些规则检测，可能漏检。
  - 建议：把风险检测移到服务端，或为检测接口返回完整可执行规则但不暴露敏感维护字段。

- [ ] 增加服务端输入校验。
  - 现状：企业和报告 API 主要保存客户端传入的 JSON payload。
  - 建议：为企业、报告、规则、AI 请求分别定义 schema，校验字段类型、范围、必填项和长度。

- [ ] 确认税务规则的专业口径。
  - 现状：规则适合 MVP 演示，但缺少政策版本、地区差异、适用范围、复核人和发布时间。
  - 建议：上线前建立规则发布流程，至少包含专业复核、依据链接、版本号和变更记录。

- [ ] 明确 DeepSeek 数据合规边界。
  - 现状：AI 接口会把企业资料和风险内容发送到第三方模型服务。
  - 建议：补充用户授权、隐私政策、数据脱敏策略、供应商合规评估和 AI 失败降级说明。

## P1：上线前强烈建议完成

- [ ] 增加组织 / 租户模型。
  - 现状：只有 `owner_user_id` 级别隔离。
  - 建议：支持组织、成员、角色、企业归属、邀请、移交和离职处理。

- [ ] 补齐部署配置。
  - 现状：仓库缺少 `wrangler.toml`、环境变量样例、预发 / 生产环境说明和自动部署配置。
  - 建议：新增 Cloudflare Pages / D1 / DeepSeek 的标准配置说明与 `.env.example`。

- [ ] 工程化 D1 迁移。
  - 现状：迁移依赖手动执行 SQL，部分 `ALTER TABLE ADD COLUMN` 不可重复执行。
  - 建议：引入可追踪、可回滚、可分环境执行的迁移流程。

- [ ] 收紧健康检查接口。
  - 现状：`/api/health` 未鉴权且返回数据表总量。
  - 建议：公网只返回最小健康状态，详细指标进入内部监控。

- [ ] 补充审计日志。
  - 现状：审计主要覆盖部分管理员动作。
  - 建议：记录企业新增 / 修改 / 删除、报告生成 / 删除、AI 调用、导出、管理员代入等关键动作。

- [ ] 增加 API 集成测试。
  - 现状：测试主要覆盖规则引擎。
  - 建议：覆盖认证、用户隔离、管理员权限、规则 CRUD、报告 CRUD、AI 失败降级。

- [ ] 建立报告交付链路。
  - 现状：Word 导出由浏览器生成，缺少版本留痕和服务端文件管理。
  - 建议：服务端生成报告，保存到 R2，记录版本、模板版本、下载权限和导出日志。
